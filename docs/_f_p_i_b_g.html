Logging

We log all major events for success and failure with a debug verbosity level. The level is from 0 to 5, 5 being the most verbose.

Logging succesfull function calls.

Verbose level of 0 is for any successful opening of a file or communications port, testing a directory for existence, or any other major function call. Even though these are succesfull we need to see a sort of logging stack when we review the log file. It is really important to log as much as you can.

Also, log reads and writes but set the verbosity level to 5 since this will usually flood the log file and needs to be used in rare cases.

# This is the logging file line for success. It requires inspect
import inspect
self.log( 1, # The debug verbosity level.
inspect.currentframe().f_lineno, # The calling line number - automatic.
__file__, # The file from which it is called - automatic.
inspect.currentframe().f_code.co_name, # Object creation name - automatic.
self.ObjName, # The name assigned to the object - automatic prob. redundant.
0, # The error code is 0 for successful calls.
"Function success") # The error text describing in detail what succeeded.

Logging failures of function calls.

Verbose level of 0 is for any failure to open a file or communications port, testing a directory for existence, or any other major function call. There are three parameters of this call that are not aotuomatic. The debug level which for errors is 0, the error number and text. The error number is formed by adding numbers sequentially to the class error level set as self.log.lvl in the class definition. The table in The text must provide enough detail to determine why the function failed.

# This is the logging file line for failure. It requires inspect
import inspect
self.log( 1, # The debug verbosity level.
inspect.currentframe().f_lineno, # The calling line number - automatic.
__file__, # The file from which it is called - automatic.
inspect.currentframe().f_code.co_name, # Object creation name - automatic.
self.ObjName, # The name assigned to the object - automatic prob. redundant.
self.dlvl+1, # The error code. Add numbers sequentially to the error level for this class.
f"Running Mod:" + modName) # A descriptive string. Include all of the information needede to solve the error.

Exercising and Unit testing

There is a difference between Exercising and Unit Testing.

Exercising

Exercising code means writing code that performs a specific function. This code resides in the test directory. There are three reasons for this.

  • To push developers to write code modualarly, in small bites
  • To have example code avaiable for new developers who may need to extend what the original developers have done.
  • So that the code reviewer can review the developer tasks more easily.

When you become project mangers you will understand and appreciate the importants of this approach.
